# 飞书项目（Meego）接入

飞书项目是 Case Flow 当前维护的标准需求与缺陷平台，但不是产品启动前提。未配置飞书时，部署方仍可使用 `/quick` 作为独立的 Markdown Case 打磨工作台，完成导入、规则打标、查看、编辑和导出；缺少的是项目池、需求目录同步、需求绑定和飞书 Bug 闭环。详见 [快速模式](快速模式.md)。

把飞书项目里的"需求"工作项拉成 Case Flow 的来源项目（`requirement_pool`），再由人工纳入一级目录形成二级需求。

状态：**已完成真实飞书项目环境验证**——空间拉取、用户同步、负责人/状态/迭代解析、卡片展示与筛选全链路可用。公开仓库不包含验证时使用的空间标识和业务数据。

## 结构对应

| 飞书项目 | Case Flow |
|---|---|
| 空间 project_key（= 一个部门） | 来源维度（按空间拉取/筛选），存 `requirement_pool.source_space` |
| 工作项类型 `story`（需求） | 二级需求来源 |
| work_item_id（如 `<work_item_id>`） | `requirement_pool.external_key`（去重/更新键，也是卡片编号 + 飞书跳转 id） |
| 名称 | 标题 → 二级需求 title |
| 状态 state_key（如 `state_testing`） | 经空间 `status_in_testing_state_keys` 判定是否"测试中"；原值存 `external_status` |
| 角色 role_id（QA/前端/后端） | 经空间 `role_map` 映射到 tester/frontend/backend；tester→负责人 + users 表 |
| 迭代排期 | 经 `default_sprint_field` / 空间 `sprint_field` 取 id，再 `sprint/query` 解析名称（含排期区间） |
| 版本 | **手动指定**（纳入时填，组内唯一），不再从标题正则解析 |
| 创建时间 created_at | 卡片展示 + 池/目录排序（倒序） |

> 一级目录手动建、手动把来源项目纳入；不删除——外部缺失也不动已沉淀数据（见 `产品范围.md`）。
> **锚点：只拉「QA(测试人员)已绑定」的需求**；没绑测试的看板不入池。

## 鉴权（plugin 模型）

飞书项目 OpenAPI 用**插件鉴权**：`plugin_id + plugin_secret` → 换 `plugin_token`（`POST {siteDomain}/open_api/authen/plugin_token`，body `{plugin_id, plugin_secret, type:0}`）→ 调业务 API。

**实测要点**：
- 工作项类 API **必须额外带 `X-USER-KEY`**——只带 `X-PLUGIN-TOKEN` 会报 `20039 Plugin Token Must Have User Key`。
- `user_key` = 某有权限用户的 key（可用飞书 MCP `search_user_info` 解析）。
- 旧 secret 失效报 `10204 wrong plugin secret`——被后台重置后旧值即作废。

## 已验证端点（直连，backend 用）

- `POST /open_api/authen/plugin_token` — 取插件 token。
- `POST /open_api/{project_key}/work_item/filter` — 工作项列表，body `{work_item_type_keys:[story], page_num, page_size, created_at:{start,end}}`。**`created_at` 区间是服务端过滤**；数据量较大的空间必须按时间收窄。
- `POST /open_api/{project_key}/work_item/filter` + `work_item_ids` — 按用户粘贴的飞书详情页链接实时读取单条工作项。不要依赖 `GET /open_api/{project_key}/work_item/{type}/{id}`；实测该路径在飞书项目插件接口下返回路由层 `404 page not found`，不是业务 JSON 错误，也不是权限/数据不存在。
- `POST /open_api/user/query` — `{user_keys:[...]}` → `name_cn/email/avatar_url/out_id`。
- `POST /open_api/{project_key}/work_item/sprint/query` — `{work_item_ids:[...]}` → 迭代名（如"迭代A（26.06.16~26.06.29）"）。
- 角色 role_id↔名 无可用在线端点（404）→ 用飞书 MCP `list_workitem_role_config` 离线查，写进配置。

## 配置（两层）

**凭证（敏感）→ env**（`backend/.env`，前缀 `CASE_FLOW_`）：
```
CASE_FLOW_FEISHU_PROJECT_SITE_DOMAIN=https://project.feishu.cn
CASE_FLOW_FEISHU_PROJECT_PLUGIN_ID=...
CASE_FLOW_FEISHU_PROJECT_PLUGIN_SECRET=...
CASE_FLOW_FEISHU_PROJECT_USER_KEY=            # 工作项/用户查询需带 X-USER-KEY
```

**非密钥配置 → `backend/config/feishu_project.json`**：
- 部署方真实空间码、role_id 和 state_key 属于本地配置，**已 `.gitignore` 忽略**。仓库只提交模板 `feishu_project.example.json`（带假数据 + 逐字段说明）；**首次使用复制 example → `feishu_project.json` 再填真实值**。
- `pull.created_after`：拉取时间起点（服务端过滤，"全拉"的唯一时间闸门）。`pull.max_items`：每空间单次上限（0=不限）。
- `default_sprint_field`：全局默认迭代/排期字段 key。飞书项目标准通常是 `planning_sprint`；空间不写 `sprint_field` 时自动使用它。
- `spaces[]`：要拉的空间，每空间填：
  - `role_map`：`{tester:[role_id…], frontend:[…], backend:[…]}`。**值是数组，可多个 role_id 合并成一个概念并去重**（如 iOS/安卓/前端三角色都算 frontend）。tester=测试人员=顶部"当前用户"身份的唯一来源；缺某概念留空 `[]`。
  - `status_in_testing_state_keys`：该空间"测试中"对应的 `state_key`（可多个）；命中才进"我的进行中任务"。留空=先都算测试中。
  - `sprint_field`：可选，覆盖全局 `default_sprint_field`。某空间确实没有迭代概念时显式写 `null`，卡片不显示迭代、也无迭代筛选。
- **纯映射、不靠名字猜**：只维护"我们的标准 → 他们的码"。`role_id` 用 MCP `list_workitem_role_config` 查，`state_key` 去飞书后台工作流配置查（或用数据反推，见下）。
- 换公司/换空间：只改这个文件 + `.env`，不改代码。

## 拉取与展示

`POST /api/v1/sources/feishu-project/pull[?project_keys=...]`（前端"飞书项目"页选空间→拉取/全拉）只启动后台任务并返回 `jobId`；前端用 `GET /api/v1/sources/feishu-project/pull-jobs/{jobId}` 轮询进度。后台任务的业务规则不变：
1. 按 `created_at` 服务端过滤分页拉工作项。
2. 解析每条的 role_owners；**无 tester(QA) 负责人的直接跳过**（返回 `skipped_no_qa` 计数）。
3. 同步 tester 负责人进 `users`（含 `feishu_user_key/email/avatar_url`）；前后端人名只查不建用户。
4. 解析迭代名；把展示数据打包成 `source_payload._card`（编号/状态/角色人名/迭代/创建时间/飞书链接）+ `source_payload._people`(人 key→姓名，供 bug 经办人 chips 显示)，列表接口直接读。
5. upsert 进 `requirement_pool`（**没有则新建、相同则更新、拉不到但已有则保持现状，不删**），存 `source_space/external_status/owner_user_id/source_payload`(全量)。
6. **单空间无权限/查询失败（如 `10301 Check Token Perm Failed`）自动跳过、不中断整体**，在返回的 `spaces[]` 里标 `error`。
7. **实时同步已挂的二级需求**：被拉回且仍有 QA 的需求，会把挂在它上面的二级需求的**状态/标题/负责人(QA)**与飞书对齐——负责人按当前 QA 全量对齐(新增缺的、移除已不在的=换测试就转移)；**case 不动、version 不动**。飞书里 QA 全清空的需求会在第 2 步被跳过，本地既有数据不追溯清理。

**多 QA / 负责人(同步真相源=飞书)**：一个需求可能有多个 QA，全部收下；纳入目录建二级需求时为**每个 QA 各挂一条** `requirement_assignees`，几个测试人首页都能看到、按测试集软分工；之后每次重拉自动对齐（见上 7）。`owner_user_id` 仍留作"主负责人"展示。未上线阶段不承诺旧数据兼容，开发期旧脏数据直接删库重拉。

前端"飞书项目"页：卡片展示编号(工作项id)/状态/测试·前端·后端负责人/迭代/创建时间/飞书↗；支持**空间 / 测试人员(参与即匹配) / 迭代排期**三种筛选；按创建时间倒序。项目池列表和 Case 资产页左侧需求目录都走服务端筛选/分页，避免页面进入时把全部已拉取数据一次性搬到前端。

> 提交 bug 到飞书（issue 字段映射、两段式建单、飞书 OpenAPI 关键约定、换空间 SOP）见 `提交bug与飞书交互约定.md`。

## 映射示例

- 角色：QA=`role_tester`、前端=`role_frontend`、后端=`role_backend`。实际 role_id 使用 `list_workitem_role_config` 查询后写入本地配置。
- "测试中" state_key 示例为 `state_testing`。不同空间的状态码完全不同，应从工作流配置或工作项返回数据确认，不能在代码里猜测。
- 迭代字段 = `planning_sprint`，来自全局 `default_sprint_field`；空间可用 `sprint_field` 覆盖或显式关闭。
- 进"我的进行中任务"：纳入二级需求时 `lifecycle_for(space, external_status)` 命中"测试中"码 → lifecycle=测试中；首页按 当前用户(QA) + lifecycle=测试中 过滤。

## 代码位置

- `backend/app/services/sources/feishu_project.py`
  - 配置：`FeishuSourceConfig`/`SpaceConfig` + `load_source_config`（读 JSON）、`list_configured_spaces`、`lifecycle_for`。
  - 纯函数：`map_work_item`、`_extract_role_owners`、`extract_sprint_ids`、`build_card`（均有单测 `tests/test_feishu_mapping.py`）。
  - 网络：`FeishuProjectClient`（`filter_work_items`/`query_users`/`query_sprints`）。
  - 编排：`pull_into_pool`（拉取+过滤+用户同步+迭代解析+卡片+upsert）、`_sync_users`。
- 路由：`GET /sources/feishu-project/spaces`、`POST /sources/feishu-project/pull`、`GET /sources/feishu-project/pull-jobs/{job_id}`。

## 已知坑

- 旧插件 secret 会失效（`10204`）——用当前有效凭证。
- 凭证是密钥：只进 env，**绝不进前端/Git**；`feishu_project.json` 已 gitignore。
- `work_item/filter` 不带 `created_at` 会翻全量历史；数据量较大的空间务必配置 `created_after`。
