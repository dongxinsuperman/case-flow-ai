# 数据模型

数据库：PostgreSQL。ORM：SQLAlchemy（async）。结构变更**一律走 Alembic 迁移**（`backend/alembic/versions/`），不要手改表。

模型定义在 `backend/app/models/`：`requirements.py`（需求体系/用户）、`case_assets.py`（正式用例资产与执行）、`quick.py`（快速模式临时 session 与 quick 专属执行/bug/修复）、`agent.py`（助手单会话与工具调用记录）。

## 整体关系（从上到下）

```text
requirement_pool（外部项目池）
   └─(1:1)→ requirement_items（二级需求） ──(N:0/1)→ requirement_groups（一级目录，挂 functionMap）
                     │
                     ├─ import_batches（测试集批次） ──< case_assets（用例）
                     │                                      ├─(1:1) case_bodies（标题/前置/步骤/预期）
                     │                                      ├─< case_steps（拆分步骤）
                     │                                      ├─(1:1) case_work_items（执行态：状态/执行器/报告/bug链接…）
                     │                                      ├─< case_repair_drafts（诊断/修复草稿，含 diagnosis_snapshot）
                     │                                      ├─(1:1) case_bug_drafts（提交bug预填草稿）
                     │                                      └─< aiphone_execution_items
                     └─ requirement_assignees（需求负责人，关联 users）

aiphone_execution_batches（同构端执行批次：AI Phone / AI Web / AI API） ──< aiphone_execution_items（每 case×平台一条）

quick_sessions（快速模式 session，独立于正式需求体系）
   ├─< quick_cases ──(1:1) quick_case_bodies
   │                  ├─< quick_case_steps
   │                  ├─(1:1) quick_case_work_items
   │                  ├─< quick_repair_drafts
   │                  └─(1:1) quick_bug_drafts
   └─< quick_execution_batches ──< quick_execution_items

users ──(1:1) agent_sessions（助手单会话）
           ├─< agent_messages（用户/助手/result 消息）
           ├─< agent_dispatches（运行时已注册的 Agent 工具调用）
           └─< agent_bug_submissions（助手自然语言提 bug 快照）
```

## 需求体系（`models/requirements.py`）

### requirement_pool — 外部项目池（技术缓存）
| 字段 | 说明 |
|---|---|
| id | PK |
| external_key | 来源项目唯一键（unique） |
| title / description | 项目名 / 描述 |
| source_type | 来源类型，默认 `mock` |
| status | 历史状态字段；项目池展示状态不以它为事实源 |
- 业务口径：飞书项目就是二级需求。同步入池后应同时存在 1:1 `requirement_items`；未上线阶段不承诺旧数据兼容，开发期旧脏数据直接删库重拉。
- 项目池展示状态通过 `requirement_items.group_id` 现算：为空 = 未进入目录；非空 = 已归属目录。正常飞书拉取不应出现 pool 有而 item 无的业务状态。

### requirement_groups — 一级目录
| 字段 | 说明 |
|---|---|
| id | PK |
| name | 目录名（unique） |
| status | 默认 `active` |
| **function_map_files** | JSONB，`[{"filename","content"}]`（迁移 `0002`）。**[当前·旧·已无生产读者]**：批次执行、提 bug、修复和助手对话都读取新挂载编译结果，不再读取它；仅剩孤儿上传接口（`workbench.py` 的 `/requirement-groups/{id}/function-map`）仍能写入该列，该列和孤儿接口可择机退役。新管理走 Function Map 资产库 + 挂载，见 [Function Map](Function-Map.md) |
- 1:N `requirement_items`。

### requirement_items — 二级需求
| 字段 | 说明 |
|---|---|
| id | PK |
| group_id | 可空 FK→requirement_groups（`ON DELETE SET NULL`）；为空表示未归属二级需求 |
| pool_id | FK→requirement_pool（unique），飞书来源下与外部项目池 1:1 |
| title / description | 需求标题/描述 |
| status / lifecycle_status | 状态 / 生命周期；纳入时按来源空间状态码映射判定（命中“测试中”才进首页任务，见 `feishu_project.lifecycle_for`） |
| **version** | 手动指定的版本（迁移 `0005`）；**同一 group 内唯一**（唯一索引 `ux_requirement_items_group_version`）。已归属目录时必填，可编辑；未归属时为空 |
| **auto_discovery_enabled** | BOOL，NOT NULL，默认 `true`（迁移 `0028`）。自动发现开关（检查点 4），首页任务块可开关；[当前] 仅持久化开关意图，尚未接入执行/发现 |
| **test_window_start / test_window_end** | 预留的测试时间区间（可空，迁移 `0003`）；**当前不展示、不参与筛选**，未来或由飞书返回 |

### users / requirement_assignees
- `users`：id、name(unique)、display_name、status、**feishu_user_key(unique)**、**email**、**avatar_url**（迁移 `0004`）。用户由飞书拉取时同步落地，`feishu_user_key` 为稳定唯一身份（同名不同人时 name 追加 user_key 尾号）。已无 mock 用户。
- `requirement_assignees`：(requirement_item_id, user_id) 复合主键 + role；首页按当前用户经此表找需求。**一个需求可多行=多个 QA**。
- **写入时机**：① 飞书项目同步入池时立即生成未进入目录的二级需求，并按飞书该需求的**全部 QA** 各挂一条；② **每次重拉自动对齐**：本次拉回且仍有 QA 的需求，负责人随飞书当前 QA 全量同步（新增缺的、移除已不在的），二级需求状态/标题一并跟飞书更新；case/version 不动。飞书里 QA 全清空的需求会被拉取阶段跳过，本地既有数据不追溯清理。

### requirement_pool 来源字段（迁移 `0004`）
- `source_space`：来源飞书空间(project_key)；`external_status`：来源工作流 state_key；`owner_user_id`：解析出的 QA 负责人；`source_payload`(JSONB)：飞书工作项全量原始数据留存。

## 用例资产与执行（`models/case_assets.py`）

### import_batches — 测试集批次
唯一约束 `(requirement_item_id, source_name)`（同一二级需求下，来源文件名相同才视为同一导入批次并触发二次导入碰撞；`suite_title` 是展示标题，不参与导入身份判断）。
- 字段：suite_title、source_name、version、project_name、feature_name、requirement_item_id(FK)、case_count、raw_metadata(JSONB)。
- `id` 作为测试集批次 `batchId` 暴露给前端，用于 Case 页/脑图分组、导出和删除；`suite_title` 允许重复。
- 1:N `case_assets`（删除级联）。

### case_assets — 用例（资产层）
| 字段 | 说明 |
|---|---|
| id / batch_id(FK) / ordinal | 主键 / 所属批次 / 序号 |
| suite_title | 测试集标题 |
| path_nodes | JSONB，Markdown 动态路径层级事实源：`[{level,label,rawText,displayText}]` |
| module_name / product_feature / test_feature | 旧三列兼容字段，仅从 `path_nodes` 前三层回填，不作为动态层级事实源 |
| raw_title / clean_title | 完整标题正文 / 兼容镜像；两者都保留完整正文，不做去标签处理 |
| scenario_tags | `TEXT[]`，场景标签（case_tagging 产出） |
| manual | 是否人工执行类型 |
| status | 资产状态，默认 `imported` |
| version / project_name / feature_name | 元信息 |
| **source_requirement_item_id** | FK→requirement_items，**执行时按它解析一级目录的 functionMap** |
- 关联：1:1 `case_bodies`、1:N `case_steps`、1:1 `case_work_items`。
- Case 资产页手动新增 Case 不新增表：仍写入当前 `import_batches` 下的 `case_assets`、`case_bodies`、`case_steps`、`case_raw_nodes` 和 `case_work_items`。新增时只能选择该测试集已有完整 `path_nodes`，旧 Case 不重排；新增工作项复用“变更待确认”语义。

### case_bodies — 用例正文（1:1）
case_id(PK,FK) + goal、preconditions、steps_text、expected_result（都 NOT NULL）。

### case_steps — 拆分步骤
case_id(FK) + step_order + step_text。

### case_raw_nodes — 原始导入快照
case_id(FK) + raw_payload(JSONB)。

### case_work_items — 执行态（1:1，首页主要读写对象）
| 字段 | 说明 |
|---|---|
| case_id | PK,FK |
| assigned_user_id | 负责人（SET NULL） |
| **execution_status** | `not_run`/`running`/`passed`/`failed`（“最后一次覆盖”，不主动轮询） |
| **coverage** | JSONB 覆盖标记：按泳道存三态 `{lane: "passed"\|"failed"}`（未执行不落键）。泳道：app=`android/ios/harmony`、web=`chrome/safari/firefox`；AI Web 外部 `webkit` 在 case-flow 内统一映射为 `safari`。**纯展示提醒，与 execution_status 解耦，不参与流转**；仅由覆盖接口/AFN 顺带写改动，三个内容变更触发器（编辑/碰撞/自动修复）清空。详见 `端覆盖标记方案.md` |
| lifecycle_state | 待验证/已固化/待人工干预等 |
| attention_reason | 关注原因 |
| case_type | `auto`/`manual` |
| **execution_target** | 执行器：`app`(AI Phone)/`web`/`api`/`manual` |
| tag_source / tag_reason / tag_confidence | 标签来源/原因/置信度 |
| report_url / failure_type / failure_summary | 报告与失败信息 |
| **bug_url / bug_external_id** | 已提交到飞书的 bug 链接 / 工作项 id（迁移 `0007`）；提交后回写，改状态/重跑/采用修复/编辑时清空 |
| active_execution_batch_id / external_submission_id | 当前 AI Phone 批次 / submission |
| execution_started_at / execution_finished_at | 执行时间 |

### case_repair_drafts — 诊断/修复草稿
case_id(FK) + status + model_name + original_steps/proposed_steps/reason + case_snapshot/report_snapshot/gate_snapshot/analysis_trace/raw_payload(JSONB) + **diagnosis_snapshot**(JSONB,迁移 `0006`：reason/evidence/key_image/repair_channel/proposed_*/process 等) + expires_at。
- 失败有报告时**后台自动生成**(`auto_diagnose_case`)，秒开；编辑/重置/采用修复/改状态时会被清空。

### case_bug_drafts — 提交 bug 预填草稿（1:1，迁移 `0008`）
case_id(PK,FK) + title + description + editable_fields(JSONB：复刻飞书建单表单的字段列表，含模型预填的选项) + created/updated_at。
- 诊断完成后**后台预生成**(`precompute_bug_draft`)，点"提交 bug"秒开；同 repair 草稿一样随状态变化清空。

### aiphone_execution_batches — 同构端执行批次（AI Phone / AI Web / AI API）
| 字段 | 说明 |
|---|---|
| id / submission_id / submission_name | 批次 / 外部执行器返回的 submissionId；AI API 用 `local-aiapi-*` 本地 submissionId |
| requirement_item_id | FK（SET NULL） |
| **callback_token**(unique) | AI Phone / AI Web 回调路由用；AI API 仅占位，不对外暴露 callback |
| executor | `ai_phone` / `ai_web` / `ai_api` |
| status / started_at / finished_at / summary_report_url | 批次状态与汇总 |
| raw_request / raw_response / raw_callback / raw_submission | 原始报文(JSONB) |

唯一约束：`(executor, submission_id)`，避免不同执行器返回相同 submissionId 时撞库。

### aiphone_execution_items — 批次内每个执行单元
唯一约束 `(batch_id, external_case_id, platform)`。
- batch_id(FK)、case_id(FK)、external_case_id（`cf-{caseId}`）、platform、state、status_reason、run_id、report_url、device_alias_pool(`TEXT[]`)、raw_item(JSONB)。
- AI Phone / AI Web 回调按 `(callback_token/submission_id) → batch → (external_case_id, platform) → item → case_work_item` 落状态。
- AI API 由本进程后台直接写 `platform="api"` 的 item，报告落在 `/media/aiapi_reports/*.html`，再回落 `case_work_item`。

## 快速模式（`models/quick.py`，迁移 `0016`）

快速模式只服务“导入 Markdown → session 内打磨 → 导出 Markdown”。它不写正式 `requirement_items`、`import_batches`、`case_assets`、`case_work_items`、`aiphone_execution_*`。

### quick_sessions
| 字段 | 说明 |
|---|---|
| session_id | 高熵字符串主键，用户拿到即可接力 |
| source_name / suite_title | 来源文件名 / 当前测试集标题 |
| function_files | JSONB，session 临时 function 文件。**[当前·旧·停用]**：快速模式已改读 `function_map_quick_mounts`（迁移 `0030`），执行/提 bug/修复/助手 quick 不再读此列；保留列、不自动迁移，可择机退役 |
| feishu_requirement_url / feishu_target | session 级飞书需求链接与实时读取的目标快照，只服务 quick 提 bug |
| feishu_bug_url | session 级旧兼容字段；新 quick 提 bug 流程不再依赖飞书缺陷参考链接 |
| current_user_id | 提 bug 时选择的提交人，可空 |

删除 `quick_sessions` 会级联删除该 session 的 quick case、work item、修复/bug 草稿、quick 执行批次；外部 AI Phone submission 和已创建飞书 bug 不回删。后续孤儿回调由 quick callback no-op 处理。

### quick_cases / quick_case_bodies / quick_case_steps
结构对应正式 case 的轻量副本，但没有二级需求、一级目录、批次碰撞等正式字段。

- `quick_cases.path_nodes`：快速 Markdown 最后四级之前的动态路径。
- `quick_cases.core_nodes`：标题/前置/步骤/预期的原始前缀信息，用于导出时保留 `测试标题：` 等格式。
- `quick_case_bodies`：标题目标、前置、步骤、预期。
- `quick_case_steps`：步骤拆分。

快速模式只允许编辑标题、前置、步骤、预期；测试集标题和路径层级由导入 Markdown 决定，编辑页只读展示，不允许修改；不允许新增、删除、拖拽排序。

### quick_case_work_items
quick 专属执行态，字段语义与正式 `case_work_items` 基本一致，但只在 quick session 内有效。编辑 case、重跑、应用修复都会清理报告、失败信息、bug 关联和覆盖标记。

### quick_execution_batches / quick_execution_items
quick 专属同构端执行表：

- external_case_id 使用 `qcf-{quickCaseId}`，避免和正式 `cf-{caseId}` 混淆。
- AI Phone / AI Web callback 路径是 `/api/v1/quick/aiphone/callback/{token}` 或 `/api/v1/quick/aiweb/callback/{token}`；AI API 不对外 callback，由本进程后台任务直接回写。
- `quick_execution_batches.executor` 区分 `ai_phone` / `ai_web` / `ai_api`，唯一约束为 `(executor, submission_id)`。
- quick AI API 使用 `local-quick-aiapi-*` 本地 submissionId，固定 `platform="api"`，报告落在 `/media/aiapi_reports/*.html`。
- session 清理后 quick 执行批次被删除，后续外部回调 no-op。

### quick_repair_drafts / quick_bug_drafts
quick 专属诊断修复草稿与 bug 预填草稿。它们使用 quick case、快速会话从资产库选中的 Function Map（`function_map_quick_mounts`，经 `compile_quick_context` 编译）、quick 飞书需求目标，不读取正式一级目录 functionMap，也不读取正式 `requirement_pool.source_payload`。旧 `quick_sessions.function_files` 已停用不再读。
诊断修复的 `process` 记录后台编排过程：`anchor`（失败锚点/需求来源）、`root_cause`（根因轮次，最多 3 轮，按 `need_more` 才补滑窗/截图）、`repair`（仅可修时生成候选）、`backend_gate`（后台门禁）。因此 `process.length` 表示阶段条目数，不等于模型固定轮数。

## OS Agent 对话模式（`models/agent.py`，迁移 `0022`）

OS Agent 是全局浮窗里的轻量单会话能力，不写正式 case 资产、不写 quick session，也不把工具结果塞回现有执行工作台。

### agent_sessions
| 字段 | 说明 |
|---|---|
| id | PK |
| user_id | FK→users，唯一；一个用户只有一个助手会话 |
| title | 当前固定为 `OS Agent` |
| default_tool / default_resource_pool / function_context | 预留默认工具、资源池和上下文 |
| bug_target | 当前会话绑定的飞书需求快照，服务自然语言提 bug |
| pending_action | 缺硬前置时暂存的动作，例如用户先说提 bug、助手等待需求链接 |
| created_at / updated_at | 创建/更新时间 |

### agent_messages
append-only 消息流，`role` 可为 `user`、`assistant`、`result`。UI 上 `result` 仍按助手回复展示。
- `content`：自然语言内容。
- `dispatch_id`：可空，关联某次工具调用。
- `attachments`：JSONB，统一承载用户图片、关键截图、报告链接、bug 链接、工具结果摘要等。用户上传图片落在 `/media/agent_uploads/`。
- 用户消息的 `attachments.context_ref` 可记录本条消息的工作台引用，例如标准模式 `{mode:"standard", requirement_item_id, use_current_function_map}` 或 quick 模式 `{mode:"quick", quick_session_id, use_current_function_map}`。这里不保存 functionMap 正文，只保存引用和开关。

### agent_dispatches
每次工具调用一条记录，当前覆盖 `aiphone_dispatch`、`aiweb_dispatch`、`aiphone_resource_probe`、`aiweb_resource_probe`、`aiapi_run` 和 `bug_submit_feishu`。
- 外部执行器使用 `callback_token` 接收 AI Phone / AI Web 回调。
- AI API 是本进程后台执行，完成后写 result 消息和报告链接。
- CLI 只是一种可选 Tool。开源版本不内置公司的 CLI Tool；部署方自行增加的任何业务工具都必须自行负责鉴权、参数校验、审计和安全边界。
- 助手 functionMap 是消息级实时解析：工具 dispatch 时按 `context_ref` 读取当前 functionMap，AI Phone / AI Web 注入 `functionMapContext`，AI API 注入 `function_map_context`。取不到时不注入，不回落全局 env。
- 状态独立于正式 `aiphone_execution_*` 和 `quick_execution_*`。

### agent_bug_submissions
助手自然语言提 bug 的快照表，用于追溯本次由对话生成的 title、description、目标需求、提交状态和飞书 bug 链接。它不是强确认表单；缺需求链接、提交人等硬前置时在对话里追问。

### function_map_assets — Function Map 资产（全局可复用上下文卡片）
| 字段 | 说明 |
|---|---|
| id | PK |
| title | 标题，**全局唯一**（迁移 `0029` 唯一索引 `ux_function_map_assets_title`）；服务层预检查友好报错，DB 约束兜住并发竞态 |
| description | 适用场景（旧称“解释”），自动发现阶段的核心判断字段 |
| content | 正文，`NOT NULL`；列表 / 挂载接口不返回，详情 / 查看时单独取 |
| targets | `TEXT[]`，适用端 app/web/api 多选，复用 `execution_target` 语义 |
| source_type / source_filename | 来源类型（首版 `local_import`）与导入文件名 |
| created_at / updated_at | 时间戳 |
- 首版正文只由本地文本导入生成；标题 / 适用场景 / 适用端可在线编辑，正文只能导入覆盖。

### function_map_group_mounts — 一级目录挂载
| 字段 | 说明 |
|---|---|
| id | PK |
| group_id | FK→requirement_groups，`ON DELETE CASCADE` |
| asset_id | FK→function_map_assets，`ON DELETE CASCADE` |
| created_at | 时间戳；`UNIQUE(group_id, asset_id)` 幂等 |

### function_map_item_mounts — 二级需求挂载
| 字段 | 说明 |
|---|---|
| id | PK |
| requirement_item_id | FK→requirement_items，`ON DELETE CASCADE` |
| asset_id | FK→function_map_assets，`ON DELETE CASCADE` |
| created_at | 时间戳；`UNIQUE(requirement_item_id, asset_id)` 幂等 |
- 挂载是**独立引用关系**：一级目录、二级需求可各自挂同一资产，互不限制；重叠在执行编译阶段去重（接入执行为后续检查点）。

### function_map_quick_mounts — 快速会话挂载
| 字段 | 说明 |
|---|---|
| id | PK |
| quick_session_id | FK→quick_sessions(session_id)，`ON DELETE CASCADE` |
| asset_id | FK→function_map_assets，`ON DELETE CASCADE` |
| created_at | 时间戳；`UNIQUE(quick_session_id, asset_id)` 幂等 |
- 快速会话从资产库选中的 Function Map 引用（迁移 `0030`）。执行/提 bug/修复/助手 quick 都按它编译上下文。旧 `quick_sessions.function_files` 已停用（保留列、不再读、不自动迁移）。

### execution_strategy_call_logs — 执行策略调用日志（检查点 7）
| 字段 | 说明 |
|---|---|
| id | PK |
| call_id | 可追踪调用 ID（uuid），唯一；返回给前端 |
| request_group_id | 一次点击的聚合 ID（前端生成，可空） |
| mode / scope / entry | `standard`/`quick`；`single`/`batch`；submit 路由或服务方法 |
| executor | `ai_phone`/`ai_web`/`ai_hybrid`/`ai_api` |
| requirement_item_id / quick_session_id | 关联对象（按模式二选一，可空） |
| case_ids | JSONB，本次覆盖的 case |
| execution_batch_id / submission_id | 提交成功后回填 |
| trigger_user_id | 触发人（quick 取会话 current_user_id；标准暂空） |
| input | JSONB，原始输入摘要（目标执行器、设备/平台、cache/retry 等） |
| function_map_result / effective_context | JSONB，**检查点 6 填充**：发现/注入/去重/排除明细与最终上下文 |
| status | `compiling`/`submitted`/`compile_failed`/`submit_failed` |
| failure_reason | 失败原因 |
| created_at / updated_at | 时间戳 |
- 每一路 submit 落一条（迁移 `0031`）。前台建 `compiling`，后台任务成功标 `submitted`、失败标 `compile_failed`/`submit_failed`。提交前失败无外部回调，由后台把对应 case 回写 `failed`。暂无前端页面。

- `0001_initial`：初始全量表。
- `0002_group_function_map_files`：给 `requirement_groups` 加 `function_map_files JSONB DEFAULT '[]'`。

- `0021_exec_executor_unique`：标准/quick 执行批次 submissionId 改为按 executor 维度唯一。

- `0024`–`0027` Function Map 资产与挂载：`function_map_assets`（`0024` 建表）、删早期 tags 列（`0025`）、`function_map_group_mounts`（`0026`）、`function_map_item_mounts`（`0027`）。
- `0028_req_item_auto_disc`：给 `requirement_items` 加 `auto_discovery_enabled BOOLEAN NOT NULL DEFAULT true`（检查点 4 自动发现开关）。
- `0029_fn_map_title_unique`：给 `function_map_assets.title` 建唯一索引 `ux_function_map_assets_title`（标题全局唯一硬约束）。
- `0030_fn_map_quick_mounts`：新增 `function_map_quick_mounts`（快速会话从资产库选中的 Function Map 引用，双 FK `ON DELETE CASCADE`）。
- `0031_exec_call_logs`：新增 `execution_strategy_call_logs`（检查点 7 执行策略调用台账，`call_id` 唯一，按 `request_group_id`/`created_at` 建索引）。
- 新增表/列：在 `backend/alembic/versions/` 加一个 revision（`down_revision` 串好），跑 `alembic upgrade head`，并更新本文件。
