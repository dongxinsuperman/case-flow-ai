# 架构说明

## 系统定位

Case Flow 是 AI 测试执行器之上的工作台。它不替代 AI Phone、AI Web 或模型服务，而是负责管理 Case 意图、选择执行路径、接收执行证据，并把最终判断交回测试人员。

```text
飞书项目 / Markdown
        ↓
Case Flow 前端 ── Case Flow API ── PostgreSQL
                        │
                        ├── 豆包/Ark
                        ├── AI Phone
                        ├── AI Web
                        ├── AI API（进程内）
                        ├── AI Hybrid（编排层）
                        └── 飞书项目 OpenAPI / MCP
```

外部执行器负责真实执行；Case Flow 保存任务、状态、报告引用、诊断和人工审批结果。

## 部署组成

### 前端

Vue 3 单页应用，包含：

- 标准工作台：需求、目录、Case 资产、执行队列和状态脑图。
- 快速模式：不接飞书也能直接导入 Markdown。
- OS Agent：把自然语言请求路由到当前真实注册的工具。

前端只维护界面状态，不是业务状态事实源。

### 后端

FastAPI 应用负责：

- Markdown 解析、导入和碰撞审批。
- Case、需求、执行批次和回调状态编排。
- AI Phone、AI Web、AI API、AI Hybrid 的提交与反馈处理。
- 报告读取、失败诊断、修复草稿和飞书 Bug 提交。
- Function Map 资产、挂载和执行时上下文合成。

### 数据库

PostgreSQL 是业务事实源。数据库结构通过 Alembic 迁移维护，不允许依赖手工建表。

### 外部系统

- 飞书项目：当前唯一维护的需求和缺陷平台。
- 豆包/Ark：当前唯一维护的模型基线。
- AI Phone、AI Web：独立部署，通过 HTTP 提交和回调连接。
- AI API：由 Case Flow 进程内编译和执行 HTTP 测试计划。
- AI Hybrid：使用模型在已注册工具之间进行多步编排。

## 两条业务主链路

### 标准工作流

```text
飞书工作项
  → 需求池
  → 一级目录 / 二级需求
  → Markdown 导入
  → 碰撞审批
  → Case 资产
  → 执行
  → 回调 / 报告
  → 诊断、修复或 Bug
```

一级目录承载共享 Function Map；二级需求承载当前 Case 工作集。重复导入采用覆盖式碰撞审批，不维护第二套隐藏资产。

### 快速模式

```text
Markdown
  → Quick Session
  → 编辑 / 执行
  → 报告 / 诊断
  → Markdown 导出
```

快速模式不依赖飞书目录体系。未配置模型时仍可导入：先走明确标签和关键字规则，无法识别的 Case 归人工。

## 执行架构

所有执行入口最终形成“批次 + item”：

- 批次记录一次用户提交。
- item 对应具体 Case 和执行端。
- 自动 Case 提交给执行器。
- 人工 Case 进入执行中状态，由测试人员手动推进，不阻断同批次自动 Case。

AI Phone 与 AI Web 是外部回调型执行器；AI API 在进程内产生同形终态；AI Hybrid 由编排器调用多个子工具后汇总结论。

执行状态遵循“最后一次有效结果覆盖前一次”。系统不额外轮询并制造第二套状态。

## 状态与投影

数据库中的需求、Case、执行 item 和回调结果是唯一事实源。Pinia Store 和脑图只把这些状态投影到界面：

```text
数据库状态 → API DTO → Pinia Store → 脑图 / 列表 / 弹窗
```

脑图节点不能独立保存一份业务状态，否则列表、脑图和回调会发生冲突。

后端字段统一使用 snake_case，前端统一使用 camelCase；转换集中在 `web/src/api/client.ts`。

## Function Map

Function Map 是执行上下文资产，不是 Case 本体：

- 一级目录、二级需求和快速会话通过挂载表引用资产。
- 执行时按当前 Case 和执行端合成有效上下文。
- 上下文随提交发送给执行器，不提前复制成多份正文。
- 缺少必要上下文时应明确失败，不能读取隐藏全局内容兜底。

## Agent 与工具

OS Agent 只调用运行时真实注册的工具。开源版本当前注册：

- AI Phone 执行和资源查询。
- AI Web 执行和资源查询。
- AI API。
- 飞书 Bug 提交。

CLI 只是一种可选 Tool，不是 Agent 主功能。开源版本不内置公司的 CLI Tool、账号或企业数据准备 Tool；部署方可以自行增加工具。遇到未接入的工具请求时明确说明需要二次开发，不能误派给其他执行器，也不影响其他已注册工具。

## 代码分层

```text
backend/app/
  api/             HTTP 路由
  schemas/         输入输出 DTO
  models/          SQLAlchemy 模型
  repositories/    数据访问
  services/        领域编排
  report_readers/  报告证据读取
  executors/       执行器公共定义
  core/            配置与数据库

web/src/
  api/             类型化客户端
  stores/          业务状态
  graph/           脑图投影和布局
  components/      通用组件
  pages/           路由页面
```

路由层只做输入校验和响应映射；跨模型、数据库和执行器的流程放在 service 层。

## 开源边界

公开仓库包含通用流程和已维护集成，不包含：

- 真实账号、Token、空间码、字段映射或客户数据。
- 公司 CLI、SSO、账号和业务数据工具。
- 私有执行器地址、私有 CI 和企业报告格式。
- 本地数据库、截图、报告、日志和调试产物。

这些内容缺失时，系统要么继续运行与其无关的功能，要么在真正调用时明确报告“未配置”或“需要二次开发”。
