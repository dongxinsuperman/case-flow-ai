# 覆盖标记方案（Coverage Lanes）

> 在 case 卡片上增加一组**覆盖标记**：纯展示/提醒层，与执行状态彻底解耦。AFN 执行可顺带写、人工可点按改；不参与 case 碰撞、归档、诊断、bug 任何主流转。
>
> 覆盖泳道按执行器切换：`app(AI Phone) → Android / iOS / Harmony`；`web(AI Web) → Chrome / Safari / Firefox`。AI Web 外部 `webkit` 在 case-flow 内统一映射为 `safari`。同一套交互、解耦原则与重置规则，仅泳道集合不同。

状态：阶段一（前端视觉+交互）、阶段二（后端接真数据）、阶段三（多端结果可操作化）均 `[当前]` 已实现并落库。阶段三见 §9——注意它**有意演进**了阶段一/二"覆盖标记纯展示、与执行解耦"的定位。

---

## 1. 背景与问题

一条 case 的**意图**是单一的（例如"验证登录后能进入首页"），但它的**执行**天然是 `case × 端` 的：可能在 Android 跑过、iOS 没跑、Harmony 只人工验过。

现状下卡片只有一个单值 `case_work_items.execution_status`（`not_run / running / passed / failed`），它驱动卡片主色、`诊断修复`/`提交 bug`/`报告` 全套流转。这套模型说不清"这条 case 到底覆盖了哪些端"。

需求：**在不拆 case、不引入多端状态机的前提下，让卡片能如实显示各端覆盖情况**，并且这个显示既能被 AFN 自动写、也能人工手动改。

随团队对多端验证的权重逐步加大，这个"端维度可见"会越来越重要——但当前一条 case 同时压多端的频率并不高，所以第一版只做"显示"，不做"多端可操作性"。

---

## 2. 四条核心规则（方案的灵魂）

1. **系统不猜 case 应该跑几端。** 不默认要求三端都跑，只提供三个可选端位；没点亮的端只是"未覆盖"提醒，不代表失败、不构成门禁。
2. **平台状态只是覆盖提醒，不是质量门禁。** 卡片绿不代表"全端通过"，只代表"该端已确认覆盖"。
3. **端标记与执行证据解耦。** 报告 / 诊断 / 提交 bug 仍只跟 AFN 最近一次真实执行结果（`execution_status` 那条线）走，端标记不参与。
4. **手动点亮不产生证据。** 人工点绿某个端，只是"我确认过该端"，没有 run_id、没有报告，不能据此诊断或提 bug。

---

## 3. 演进过程：从"重"到"轻"

讨论早期一度考虑把"多端"做成完整的状态模型，后来逐条否掉，最终收敛到极轻的"显示层"。记录如下，避免以后有人再走一遍弯路。

### 3.1 被否决的重方案

| 方案 | 设想 | 否决理由 |
|---|---|---|
| **多端状态机** | 每条 case 维护 Android/iOS/Harmony 三套独立执行状态，各自流转 | 替用户预设"必须跑三端"，把"覆盖提醒"误做成"质量门禁"；和"系统不猜端数"冲突 |
| **总节点 + 子端节点** | 引入一个聚合"总状态"，下挂三个端子状态，聚合规则向上汇总 | 需要定义聚合/优先级规则，复杂；且违背"业务状态唯一源、脑图只是投影"的既有架构原则 |
| **影子节点 / 回笼** | 端执行各自建影子执行单元，结束后回笼合并到 case | 引入第二套状态与对账逻辑，和"执行状态最后一次覆盖、不做对账"的有意决策冲突 |
| **子表流转** | 端维度单独建表并各自承载诊断/bug 流转 | 一条 case 要同时承载多个正式流转（多端各自诊断、各自提 bug），复杂度爆炸 |

核心教训：**只要让端状态参与"判定/流转"，复杂度就会失控。** 出路是让它完全不参与逻辑。

### 3.2 收敛到的轻方案

> 不做多端状态机，只给 case 卡片加三个**可编辑的端覆盖标记**；AFN 可自动更新，人工也可手动更新。它只做视觉提醒和覆盖记录。

关键决策点（讨论中逐个拍定）：

- **三态点按循环**，不弹窗、不下拉：`未执行(none) → 通过(passed) → 失败(failed) → 未执行`。
- **重新执行 ≠ 重置**：重跑只"顺带写"它真正跑的那条泳道，其他泳道的标记纹丝不动。
- **重置只认三个内容变更触发器**：① 碰撞变更 ② case 内容编辑 ③ 自动修复落地。命中任一即把该 case 的所有泳道标记清回 `none`（语义：内容变了，旧的覆盖作废）。
- **按执行器展示对应泳道**：`app → Android/iOS/Harmony`、`web → Chrome/Safari/Firefox`、`api/manual → 不显示`。这只是**展示过滤**，不掺任何数据判断。
- **第一版先不区分来源**：绿色可能来自 AFN 自动、也可能来自人工点按；因为它不参与逻辑，是否在视觉上区分来源留待后续。

## 4. 两层语义（必须分清）

| 层 | 字段 | 含义 | 谁能改 | 参与流转？ |
|---|---|---|---|---|
| **执行证据层** | `case_work_items.execution_status`（+ report_url / failure_type / bug 等） | 当前/最近一次 AFN 真实执行结果 | AFN 回调、人工状态下拉 | 是：主色、诊断、报告、bug 全靠它 |
| **覆盖提醒层** | `coverage`（按泳道存的三态字典，如 `{android,ios,harmony}` 或 `{chrome,safari,firefox}`） | 各泳道"已确认覆盖"的备忘 | AFN 顺带写、人工点按 | 否：纯展示 |

卡片**主色仍由 `execution_status` 决定**（含执行中扫描动画），覆盖标记只是操作区里额外的几枚小图标。这样永远不会出现"卡片红但没有诊断按钮"的矛盾——证据层与提醒层各管各的。

> 注：因为两层解耦，首页统计（`not_run/running/passed/failed` 计数）继续按 `execution_status` 走，覆盖标记**不影响统计与归档**。"Android 通过、iOS 未跑"这种情况不会被解释成"全端验收通过"。

## 5. 与现有代码的契合点

方案对后端是**纯增量、低风险**，因为底子已经具备：

- **端维度数据本就存在**：`aiphone_execution_items` 表已有 `platform` 与 `state`（AFN 跑多端时每端一行、每端一个 report）。"哪个端跑了、什么结果"后端早就知道。
- **最后一次覆盖**：回调 `_apply_item_event` 把每个端终态都写进同一个 `execution_status`，后到覆盖前一个（AGENTS.md 既有决策，不做对账）。端标记正是在这之上加一层"按端记录"。
- **三个重置触发器恰好是已有的三个重置块**，各加一行清空即可：
  - case 内容编辑 → `backend/app/services/case_assets.py`（`update_case_asset`，注意现状是**任何编辑**都重置）
  - 自动修复落地 → `backend/app/services/case_repair.py`（`apply_repair_draft`）
  - 碰撞变更 / 导入 → `backend/app/services/importing.py`（碰撞命中重置为"变更待确认"）

## 9. 阶段三 · 多端结果可操作化（`[当前]` 已实现）

> 落地清单（实际改动，对应 §9.7）：
> - **回调等齐+取最差**：`executions._aggregate_case_items` / `_worst_wins`。当前批次还有端在跑→running（等齐）；否则按**每端最近一次结果（跨批次）**取最差——`list_case_platform_results` 已改为“按平台去重取每端最近一次”，所以**单独重跑一个端不会丢其它端上一次的结果**（卡片不会被错误带绿）。`_apply_item_event` 按聚合落 `execution_status`，且只在整体非失败时清诊断草稿（修掉兄弟端成功误清失败端诊断）。

> - **提 bug**：迁移 `0013` 加 `case_work_items.bugs` JSONB 列表；`bug_submit` 去掉“只提一次”闸、改追加；**多端证据**——`finalize_bug` 上传每端截图、描述附每端链接；提交体 `key_images` 让用户在弹窗里**叉掉/恢复**按需带图（None=全带、[]=不带、子集=部分带）；前端按钮“提交 bug / 已提交 N” + 弹窗顶部常驻已提交列表 + “再提一条”（重置图选择）。
> - **重新执行 = 新一轮，强制重新派生**：`submit_aiphone_execution` 清 `report_url/failure/bug_url/bugs` + `_clear_repair_drafts`（同时删诊断草稿与 bug 草稿）。因诊断是融合的，**重跑任一端都会重新融合诊断 + 重建 bug 草稿**（结果回落且整体失败时 `auto_diagnose_case` 自动重跑，`precompute_bug_draft` 重建）。⚠️ 勿删 `_clear_repair_drafts`，否则破坏此保证。

### 9.0 与前两阶段的关系（先读）

阶段一/二把覆盖标记定义为"**纯展示、与执行状态解耦、手动不产生证据**"（见 §2、§4）。当多端真实执行铺开后，那套"先失败后成功会被覆盖、失败端看不到也修不了"的痛点变成现实。阶段三据此**有意演进**：

> **端状态从"展示备忘"升级为"每端的执行结果"，并参与卡片判定。** 也就是说，§2 的规则 2/3、§4 的"解耦/纯展示"在本阶段被改写——端不再只是提醒，而是卡片状态的来源。

不变的仍然是：**一条 case、一个脑图节点、一张卡片、不拆 case、不做多端状态机**。复杂度交给 AI（融合诊断），不摊到 UI。呈现采用 **B 方案（卡片一行 + 每端标识 + 需要时浮层）**，因为它理解成本最低。

> **根基哲学（整套不打架的原因）**：系统不牵制谁——**谁最后回落状态，就是什么状态（last-write-wins）**。**手动是最高优先级的一手**：人一动手就按人的来、该清就清，不管底下有多少端、多少错误。**AFN 是另一手**：按端跑、跑完取最差。两手都服从"最后回落赢"，所以永不冲突、无需互相迁就。阶段三**只新增 AFN 多端这一手，完全不改手动那一手**（手动维持现状）。

### 9.1 起因（要解决的真问题）

多端一个批次执行：旧的"最后一次回调覆盖 `execution_status`"会让"先失败、后成功"的失败被成功**随机掩盖**（取决于回调到达顺序）；且兄弟端成功的回调会 `_clear_repair_drafts` 把失败端的诊断清掉。需要：**失败不被掩盖、每端结果可见可操作**，但仍不拆 case、不搞多端状态机。

### 9.2 端状态 与 卡片状态

- **每端状态**：未执行 / 执行中 / 通过 / 失败；失败可附带证据（报告 / 诊断料 / bug 草稿），仅当 AFN 真实产出。
- **绿（通过）与 执行中 只来自 AFN 真实执行**；"执行中"不可手动设（手动设不出"正在跑"这件事）。
- **卡片状态由两个"动手的人"决定**：
  - **AFN 每跑完一次** → 卡片 = 当前所有端里**最差**（worst-wins）：任一端红→失败；无红有在跑→执行中；无红有绿→通过；全未执行→未执行。
  - **手动 = 最高优先级 override**：左侧四态下拉 = 整卡级；点某个端 = 按端级。人设了就听人的。
- **端标识不"实时指挥"卡片**。"端红→卡红"是 **AFN 跑完时取最差**的结果，不是联动；手动设的卡片状态不会被遗留的红端拽回去。

### 9.3 执行 / 重跑（按端 vs 整卡）

- **AFN 执行 = 按端**：多端同时跑时卡片显示"执行中"，**等所有端落定**才落终态；端逐个变色。
- **重跑 = 只动被跑的端**，其余端保留；AFN 跑完按"所有端取最差"重算卡片。所以**还有端失败→卡片继续失败；全部修好才通过**。
- **手动四态下拉 = 整卡级**，保持今天行为（不加按端逻辑）。两条路天然分开（不同操作入口、不同代码路），不靠"事后分辨"。

### 9.4 重置（清所有端 + 信息）

- **覆盖标记**：仅三个**内容变更**触发重置（① 应用修复 ② 编辑 case 内容 ③ 碰撞变更）；**执行/重跑不重置覆盖标记**（只按端覆盖被跑的端的结果）。
- **bug / 诊断**：与覆盖标记不同——**重新执行就是新一轮**，会清 `report_url/failure/bug(_url)` + 删诊断/bug 草稿；因诊断是融合的，重跑任一端都重新融合诊断 + 重建 bug 草稿（见 §9 落地清单第 4 条）。

### 9.5 手动 = 最高优先级 override

手动改卡片状态或改端状态，都是人工 override，**允许任意改、信息留不留随人**（人说了算）。例：把 AFN 失败的端手动点成绿，它的报告可以仍留着，无妨——与手动改卡片状态同理。

### 9.6 三个功能按钮

- **查看报告**：单端 → 直接打开；多端 → **浮层**列各端报告（只列真有报告的端）。
- **诊断修复**：取当前**所有失败端**报告 → 合并成一份（统一编号截图 + 分段轨迹）分析 → **融合成一份**修复方案（步骤三端共享，不能按端各修）；**每端各存一张关键截图**当证据；失败后台自动跑；某端转成功 → 从诊断中剔除、后台重算；**接受修复 → 整卡重置**。
- **提交 bug**：失败时出现；**与端解耦、可多次提交**；没提过="提交 bug"，提过="已提交 N" + 弹窗顶部常驻已提交列表 + "再提一条"。**每端证据图**随描述附上，且可在弹窗里**叉掉/恢复**按需带（给不同端负责人提不同子集）。重新执行后归零（新一轮）。

### 9.7 必须配套的改造（实现要点）

- 回调改"**等所有端落定再落终态 + 取最差**"，别让先到的回调直接定终态。
- 诊断从"喂一份报告"改成"**喂当前所有失败端报告，分端 + 融合**"。
- **兄弟端成功不再清失败端诊断**。
- 多端**查看报告浮层**。
- 提交 bug 改"**链接列表 + 已提交计数 + 可多次**"。

### 9.8 工作量判断

中等、可控、不炸架构（仍是加法）。三块较重：① 回调"等齐 + 取最差"改造；② 诊断"分端 + 融合"；③ bug 多提（存储 + UI）。其余（报告浮层、不误清诊断）较轻。
