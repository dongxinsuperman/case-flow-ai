# 模型配置与维护边界

Case Flow 当前只维护和验证豆包/Ark。即使其他模型提供 OpenAI 风格接口，也不代表请求参数、Responses API、工具调用、图片输入和推理参数完全兼容。

## 配置

```env
CASE_FLOW_LLM_PROVIDER=ark
CASE_FLOW_LLM_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
CASE_FLOW_LLM_API_KEY=
CASE_FLOW_LLM_MODEL=doubao-seed-2-0-pro-260215
CASE_FLOW_LLM_MAX_TOKENS=16000
```

真实 API Key 只能放在本地 `backend/.env` 或部署环境变量中，不能提交到 Git。

## 没有配置模型时

- 后端和前端可以启动。
- 快速模式和标准模式可以导入 Markdown。
- 执行端打标先使用明确标签和关键字规则；仍无法判断的 Case 标为人工，导入成功。
- AI API 自然语言编译、AI Hybrid、失败诊断和智能修复等真正依赖模型的操作不可用，并应返回明确错误。

## 执行端打标的模型异常

执行端打标保持与上游项目一致：模型超时、网络错误、非法响应或缺少部分结果时，保留此前的规则结果，仍无法判断的 Case 按人工执行兜底并继续导入。这个兜底只用于导入时选择执行端；AI API、AI Hybrid、诊断和修复等真正依赖模型产出结果的功能不会因此假装可用。

## 其他模型

项目不承诺其他模型兼容，也不提供适配支持。使用方需要自行修改模型调用代码、验证所有使用路径，并维护相应测试。不能只替换 Base URL 就宣称兼容。
