# 执行队列与设备就绪状态方案

这是一份"先规划、后实现"的方案文档。目标是把首页执行（单条执行 + 批量执行队列）这一块做扎实，第一个要解决的具体问题是：**在选择设备时，标记出每台机器当前是否"就绪（ready，可立即调用）"还是"占用中 / 离线"。**

状态标注沿用主文档约定：`[当前]` 已实现，`[规划]` 要做，`[待定]` 待拍板。

---

## 1. 背景与目标

- [当前] 单条执行：点"执行"→ 弹出设备选择弹窗（`devicePicker`）→ 点一台在线设备 → 立即提交。
- [当前] 批量执行：勾选多个 case → 打开执行队列 → 勾选多台设备组成设备池 → 确认 → 整批 + 共享设备池交给 AI Phone，由 AI Phone 调度器排队/并发。
- [规划] 在以上两个选设备的场景里，明确标记每台设备"是否 ready（可立即调用）"，让用户知道选了之后是"立刻能跑"还是"要排队等占用释放"。

---

## 2. 现状调研结论（重要事实）

### 2.1 AI Phone 侧设备接口（`ai-phone/backend/ai_phone/server/api/devices.py`）

平台提供两个接口，语义不同：

- `GET /api/devices/available`（匿名，**case-flow 当前调用的就是它**）
  - 语义："外部调用方提交任务前，先看现在哪台能接单"。
  - **只返回此刻可立即调度的设备**：`status==online` 且 `readiness.ready` 且 `未被锁占用` 且 `agent WS 在线`，四个条件全满足才返回。
  - 返回字段：`serial / alias / platform / brand / model / osVersion / screenWidth / screenHeight / lastSeenAt`。
  - **不带任何占用/状态字段**——因为列表里的设备按定义全是空闲 + ready 的；占用中、未就绪、离线的设备直接被排除，根本不出现。

- `GET /api/devices`（全量列表）
  - 返回**所有**设备，并叠加占用信息：
    - `status`：`online / offline`
    - `lock`：占用锁信息（持有者等）或 `null`
    - `effective_status`：派生状态。**online + 被锁 → `busy`**；否则取 `status`（`online` / `offline`）。
    - `extra.readiness`：agent 上报的就绪信息（含 `ready` 标志）。
  - 注意：`effective_status` 只综合了 `status + lock`，**没有并入 `readiness.ready`**。即 `effective_status==online` 不等于 `/available` 口径的"ready"。

### 2.2 case-flow 侧现状

- 后端 `backend/app/services/executions.py::list_aiphone_devices()`
  - 只请求 `{aiphone_base_url}/api/devices/available`，原样把 `devices` 列表返回（`source / devices / error`）。
  - 因此 case-flow **当前只能看到"空闲+ready"的设备**，占用中的机器对它不可见。

- 前端 `web/src/types/case.ts::AIPhoneDevice`
  - 字段只有 `alias / serial / platform / brand / model / osVersion`（外加透传），**没有任何 ready/busy/status 字段**。

- 前端展示（`web/src/pages/HomeView.vue`）
  - 单条执行设备弹窗（`devicePicker` → `pickerDeviceGroups`）：每台只显示 别名 + 品牌/型号/系统，无状态标记。
  - 批量执行队列设备勾选（`executionQueue.devices` → `queueDeviceGroups`）：同上，无状态标记。
  - 选中的设备最终拼成 `deviceAliasPools = {平台: [alias,...]}` 提交。

> 一句话现状：**因为用的是 `/available`，现在展示出来的设备其实"个个都是 ready 的"，但既没有显式标记，也看不到占用中的机器。** 要满足"标记 ready/占用"，必须改数据来源。

## 3. 设计选型（已定）

**决策：两个接口都用。** 一个负责"空闲"，一个负责"占用中"，并起来就是要显示的"可用设备集合"。

- `GET /api/devices/available` → **空闲且就绪**的设备集合（标"空闲"）。
- `GET /api/devices` → 全量；其中 `effective_status==busy`（在线被占用、但本身可用）的 → 标"占用中"。这部分 `/available` 看不到，必须靠全量补。
- 两个并起来 = 可见集合（空闲 ∪ 占用），**都可选**。
- 既不在 `/available`、`effective_status` 也不是 `busy` 的（在线但未就绪 / 离线）→ **不显示**。

case-flow 给每台可见设备打一个 `occupancy` 标记：
- `idle`（空闲）：serial 命中 `/available`。
- `busy`（占用中）：未命中 `/available` 且 `effective_status==busy`。
- 其余（未就绪 / 离线）：直接过滤掉，不下发给前端。

> 为什么不只用全量 `/api/devices`：它的 `effective_status` 没并入 `readiness.ready`，单看会把"在线但还没准备好"的机器误当可用。用 `/available` 作为"空闲且就绪"的唯一判定，直接复用平台自己的就绪逻辑，最准。

## 4. 具体改动点

### 4.1 后端 `executions.py::list_aiphone_devices()`
- [规划] 同时请求 `/api/devices`（全量）和 `/api/devices/available`（空闲就绪 serial 集合）。
- [规划] 组装"可见集合 = 空闲 ∪ 占用"，给每台补 `occupancy ∈ {idle, busy}`；未就绪 / 离线的过滤掉不返回。
- [规划] 可选透传 `effectiveStatus`、占用方类型（`lockHolderType`，仅展示，不含 token）。
- [规划] 降级：若全量 `/api/devices` 拿不到，回退到只用 `/available`（全部当 idle），`source` 标记降级。
- [边界] 不暴露锁 token、agent_id 等内部字段。

### 4.2 schema / 类型
- [规划] `schemas/executions.py` 设备出参补 `occupancy`（与可选 `effectiveStatus`）。
- [规划] 前端 `types/case.ts::AIPhoneDevice` 补 `occupancy?: 'idle' | 'busy'`、`effectiveStatus?: string`。

### 4.3 前端展示（单条弹窗 + 批量队列两处都改）
- [规划] 每台设备显示占用徽标：`idle` → 绿色"空闲"；`busy` → 橙色"占用中"。
- [规划] 两种都**可选**（占用中只是提示，选了会进 AI Phone 排队）。
- [规划] 顶部汇总，如"空闲 3 / 占用 2"。

## 5. 已确认决策

1. **接口**：两个都用（§3）。`/available`=空闲，全量 `busy`=占用中。
2. **显示与可选**：可见集合 = 空闲 ∪ 占用，**两者都可选**。占用中仅回显状态（选了会排队，不会立刻跑）。
3. **隐藏规则**：未就绪（没准备好、不可投递）和离线设备**一律不显示**。
4. **不做实时刷新**：打开设备面板时拉一次（两个接口），显示固定；关闭再打开才重新拉。
5. **单条 + 批量两处都加**占用 / 空闲标记，规则一致。
6. **0 台可用**：可见集合为空给空状态提示；批量里有 App case 但无可用设备时确认按钮禁用 + 提示（沿用现有"至少选一台"）。
7. **不改 AI Phone**：用"两个接口求交"，case-flow 侧自洽。

## 8. 执行配置补充：重试 / 轨迹缓存 / functionMap（已实现）

### 8.1 重试 + 轨迹缓存（仅批量队列 AI Phone 块）
- 位置：批量执行队列的 AI Phone 策略块内，单条执行不提供。
- 控件：失败重试次数（默认 0）+ 轨迹缓存模式下拉 `off/v1/v2/v3`（默认 off，保留 AI Phone 原意枚举）。
- 提交：随 submission 带 `retryMax` / `cacheMode`（后端 `AIPhoneSubmitIn` 原本就支持，之前写死 0/off）。
- 边界：缓存需 AI Phone 服务端 `trajectory_cache_enabled`，否则回落 off。
- 待办：v1/v2/v3 的业务含义待补充后，把下拉文案做友好。

### 8.2 Function Map 资产库与显式挂载
- 定位：Function Map 是 App / Web / API 执行器的可复用业务上下文资产，不再只是一级目录上的 App 附件。
- 挂载：一级目录、二级需求和快速会话都通过引用显式挂载资产；一级目录挂载可被其下二级需求继承。
- 执行携带：单条和批量执行都会按本次范围读取显式挂载，按执行器适用端过滤、按资产 ID 去重，合成为提交级 `functionMapContext`。
- 多目录批次：合并本次涉及目录和需求的匹配资产，不因跨目录而静默丢弃；资产边界保留标题、ID 和来源。
- 边界：新资产模型没有 8000 字上限，也不回落全局 env；无匹配挂载时不携带上下文。
- 核对：执行流水展示本次实际提交的 Function Map。详见 [Function Map](Function-Map.md)。
- 旧接口：`/requirement-groups/{id}/function-map` 和 `requirement_groups.function_map_files` 仍为兼容遗留，当前前端和执行链不再读取。

### 8.3 明确不做
- AI Phone 鉴权：内网通用、无需鉴权（已确认）。
- 多平台同一 case 投递：暂不做（一条 case 一个平台，避免状态合并复杂化）。
