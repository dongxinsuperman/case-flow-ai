# 产品范围

## 目标

Case Flow 把 Markdown 测试用例转换成结构化的 AI 辅助测试工作台：

1. 导入 Markdown 测试用例。
2. 保留八级用例结构。
3. 重复导入时处理打磨阶段的用例碰撞。
4. 用脑图可视化当前工作用例集。
5. 将用例分发给执行器。
6. 接收执行反馈。
7. 分析失败报告并给出步骤修复建议。

## 当前建设顺序

1. Markdown 导入与持久化。
2. 首页脑图投影与 Case 操作区。
3. Case 资产编辑、移动、删除，以及父级空链路清理。
4. 打磨碰撞审批。
5. 执行器分发契约与 webhook 反馈。
6. 报告读取契约与错误修复流程。
7. 最终归档与历史资产融合。

## 明确决策

- 打磨阶段，每个二级需求只保留一份当前工作用例集。
- 打磨导入是偏覆盖的，不是历史归档合并。
- 需求目录与负责人数据是稳定配置数据；导入的 case 是可清理的工作数据。
- 完全 hash 一致的 case 在碰撞审批中直接忽略。
- 非完全一致的 case 先进入一对一决策域，再计算删除候选。
- 一个新 case 一旦锁定到某个旧 case 作为 1:1 主候选，这个旧 case 在同一次审批里不能再作为删除候选出现。
- 删除候选只来自没有 exact match、也没有被锁定进 1:1 替换域的旧 case。
- 碰撞审批必须整体提交。单条 add、replace、delete、keep、skip 都只是暂存决策，只有所有必处理项完成并确认后才写入数据库。
- 测试集删除只清理当前二级需求下的某个测试集及其全部 case，不删除二级需求、一级目录或 functionMap；外部执行的迟到回调按已删除资产忽略。
- 当前执行状态采用“最后一次状态覆盖前一次状态”。
- 生命周期可以保留在数据模型中，但在产品逻辑没想清楚前，不驱动用户交互。
- 开源版本暴露扩展点；企业适配器放在本项目之外。

## 打磨碰撞模式

打磨导入模型会相对激进，因为它描述的是未完成需求的当前工作集，不是历史归档：

- 完全相同 case：静默跳过。
- 内容变化但意图相同：用户可以替换旧 case。
- 内容变化且是新意图：用户可以作为新 case 加入。
- 新完整快照中缺失旧 case：用户可以删除或保留旧 case。
- 相似关系不确定：仍由用户决定；模型只提供候选关系和原因。

当前实现保持一对一匹配。如果一个新 case 从概念上可能替换多个旧 case，也只能锁定其中一个旧 case。剩余旧 case 会作为独立决策继续展示，不会被静默丢弃。

## 场景化说明

比如一个需求还在打磨中，第一版写的是“登录”，第二版需求变成“注册”。在打磨阶段，我们关心的是当前这个需求最终要留下什么用例，所以可以让新导入直接推动当前工作集变化。

但如果“登录”已经上线归档，后来又做“注册”，那就是最终归档/历史融合问题，不属于当前打磨碰撞逻辑。
