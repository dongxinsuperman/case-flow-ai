# 快速模式

快速模式是一套独立的 Markdown Case 打磨工作台，入口为前端 `/quick`。它不依赖飞书项目体系：可以直接导入一份测试用例，完成结构化查看、规则打标、编辑整理和导出；配置相应模型或执行器后，还可以继续执行、诊断和修复。它不要求先建立飞书项目、一级目录或二级需求。

因此，没有飞书配置不代表 Case Flow 不可用。缺少的是飞书项目池、需求目录同步、需求绑定和飞书 Bug 闭环；Case 打磨及其可选执行能力仍然保留。

## 最小配置

```env
CASE_FLOW_DATABASE_URL=postgresql+asyncpg://...
```

不需要配置：

```env
CASE_FLOW_FEISHU_PROJECT_PLUGIN_ID=
CASE_FLOW_FEISHU_PROJECT_PLUGIN_SECRET=
CASE_FLOW_FEISHU_PROJECT_USER_KEY=
```

上述空配置不会阻止快速模式启动、导入、编辑和导出。

## 使用流程

1. 启动 PostgreSQL、后端和前端。
2. 打开 `http://127.0.0.1:5173/quick`。
3. 导入符合快速模式层级规则的 Markdown。
4. 查看或编辑 Case，并按需选择 AI Phone、AI Web、AI API 或 AI Hybrid 执行。
5. 查看回调、报告和诊断；最后导出 Markdown。

## 可选能力

- 飞书项目：只在绑定飞书需求和提交飞书 Bug 时需要。
- 模型：导入不是强依赖；AI API 编译、Hybrid、诊断和智能修复需要豆包配置。
- Function Map：可以不挂载；需要业务上下文的 Case 应明确挂载，系统不会使用隐藏的全局上下文兜底。
- AI Phone / AI Web：只在选择相应执行器时需要部署。

## 缺失能力如何表现

- 未配置飞书却提交 Bug：明确拒绝并说明缺少飞书配置。
- CLI 只是一种可选 Tool。开源版本不内置公司的 CLI Tool；依赖企业测试数据准备的任务会明确说明需要二次开发，不影响快速模式的其他能力。
- 未配置模型：导入继续使用关键字规则，无法识别的 Case 归人工；诊断或 Hybrid 等真正依赖模型的操作明确失败。
- 执行器不可达：返回真实设备查询或提交错误，不自动改成人工成功。

## Markdown 示例

```markdown
- 登录验证
  - 接口能力
    - 测试标题：【API】查询登录状态
      - 前置条件：服务已经启动
        - 操作步骤：GET /api/session
          - 预期结果：返回 HTTP 200
```

快速模式最后四级固定解释为测试标题、前置条件、操作步骤和预期结果；前面的层级数量可以按导入文件动态增减，并全部作为路径节点保留。标准模式的部署级自定义规则见 [Markdown 导入与层级配置](Markdown层级配置.md)。

测试标题开头的 `【标签】` 会被复制识别为辅助元数据，但不会从标题正文中删除。脑图、Case 列表、编辑、执行和 Markdown 导出始终使用完整标题，例如 `【改造】【重点】逐层返回` 导出后仍保持原样。
